# Разработка КАР-точки

Краткая инструкция для тех, кто дорабатывает проект: настройка окружения, git, тесты и локальный запуск.

## Первоначальная настройка

1. **Склонируйте репозиторий** (или откройте папку проекта в Cursor).

2. **Создайте локальный конфиг Supabase** — файл `js/config.js` не хранится в git:
   ```bash
   cp js/config.example.js js/config.js
   ```
   Откройте `js/config.js` и впишите `SUPABASE_URL` и `SUPABASE_ANON_KEY` из панели Supabase.  
   Если ключи оставить пустыми, приложение работает только в **демо-режиме** (IndexedDB).

3. **Установите зависимости для тестов** (нужен [Node.js](https://nodejs.org/) 18+):
   ```bash
   npm install
   ```

## Локальный запуск приложения

Приложение — статические файлы, сборка не нужна.

**Вариант 1 — простой HTTP-сервер (рекомендуется для PWA и ES-модулей):**
```bash
python3 -m http.server 8080
```
Откройте в браузере: http://localhost:8080

**Вариант 2 — двойной клик по `index.html`:**  
Демо-режим обычно работает, но ES-модули и service worker могут вести себя иначе без localhost — для полной проверки используйте сервер выше.

## Тесты

Тесты покрывают алгоритмы интервального повторения (`js/srs.js`).

**Запустить один раз (CI / перед коммитом):**
```bash
npm test
```

**Запустить в режиме наблюдения** (перезапуск при изменении файлов):
```bash
npm run test:watch
```

При успехе вы увидите что-то вроде:
```
 ✓ tests/srs.test.js (8 tests)
```

## Работа с Git

### Посмотреть изменения
```bash
git status          # какие файлы изменены
git diff            # что именно изменилось
git log --oneline   # история коммитов
```

### Сохранить изменения (коммит)

```bash
# Добавить нужные файлы (js/config.js добавлять НЕ нужно — он в .gitignore)
git add .

# Создать коммит с сообщением
git commit -m "Кратко опишите, зачем это изменение"
```

Примеры сообщений:
- `fix: исправить подсчёт карточек к повторению`
- `feat: добавить офлайн-очередь для облака`
- `refactor: разбить app.js на модули`

### Отправить на GitHub (push)

```bash
# Первый раз для новой ветки
git push -u origin имя-ветки

# Дальше, если ветка уже привязана к remote
git push
```

Если работаете в `main`:
```bash
git push origin main
```

### Типичный цикл разработки

```bash
git pull                    # подтянуть свежие изменения с сервера
# ... правки в коде ...
npm test                    # убедиться, что тесты проходят
git add .
git commit -m "описание"
git push
```

### Важно

- **Не коммитьте** `js/config.js` — там ключи Supabase. В репозитории лежит только `js/config.example.js`.
- **Не коммитьте** `node_modules/` — он уже в `.gitignore`.
- Перед push желательно прогнать `npm test`.

## Деплой

Для Netlify/Vercel/GitHub Pages загружайте **всю папку проекта**, включая `js/config.js` на сервере (создайте его в настройках CI или вручную после деплоя). Без `config.js` на хостинге облачный режим не заработает, демо-режим — будет.

После обновления кода увеличивается `VERSION` в `sw.js` — так браузеры подтянут новый service worker.
